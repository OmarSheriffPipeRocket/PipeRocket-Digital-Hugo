# PipeRocket Stat Bank

A registry of every **third-party statistic** we cite across editorial content (blogs, glossary). One stat is researched and maintained **once here**, then reused/refreshed across every page that cites it — instead of re-verifying the same number inside each blog.

This file is internal and version-controlled. It is one of the PipeRocket "bank" family, kept non-overlapping:
- `stat_bank.md` (this file) — third-party external statistics we cite
- `experience-bank.md` — first-party/proprietary data + named frameworks
- `news_bank.md` — industry events/updates
- `ai_visibility_bank.md` — who gets cited/named by AI engines for our target queries

Routing: external numbers → here; proprietary numbers → Experience Bank; events → news_bank; AI citation share → ai_visibility_bank.

---

## How this bank is used

**At write time** — when a writer cites an external stat, it must either come from this bank or be **added** here. This is the single intake gate, so stale/404/recycled stats can't sneak in ad hoc.

**At maintenance time (weekly bank refresher)** — for EVERY entry, every run, check all three:
1. **Link alive?** — source URL resolves (not 404/dead). Bot-blocked 403/000 from known-protected domains (Gartner, Bain, Edelman, McKinsey, 6sense) = verify in browser, not auto-fail.
2. **Too old?** — `Published` older than ~18 months → flag `aging`.
3. **Newer figure available?** — run a web check for a newer edition/figure **every run** (not only when "due"). If found, set `Status: superseded` and fill `Replaced-by` with the new claim + source, ready to apply.

**At blog-refresh time (bi-weekly)** — the blog refresher does NO research. It looks up which entries a post references (`Referenced-by`, auto-generated by grepping source URLs across `content/`) and applies any that are flagged. Human-approved diff before commit.

## Entry schema

```
### [stat-id]
Claim: the figure exactly as we state it
Source: canonical URL
Publisher: who produced the data
Provenance: primary | secondary-recap | vendor-study   # prefer primary; flag recaps
Published: YYYY-MM (or "undated")
Cadence: annual | recurring | one-off | living-page     # metadata only; does NOT gate the newer-edition check
Status: fresh | aging | volatile | superseded | dead-link | verify-source
Replaced-by: [only when superseded] new claim + source
Referenced-by: [AUTO — regenerated by grepping Source across content/]
Note: [optional] provenance/quality/time-bound caveat
```

**Provenance tiers** (prefer higher when both fit):
- **primary** — the org that ran the study (Gartner, Forrester, CMI, Google).
- **secondary-recap** — a third party citing someone else's data ("Forrester, via X"). Replace with the primary when possible.
- **vendor-study** — a tool vendor's own blog study (Semrush, Ahrefs, HubSpot). Usable, but label it.

**Status flags:**
- `volatile` — a living benchmark that drifts (StatCounter share, Social Insider benchmarks, AI-Overview coverage %). Re-check more often; expect change.
- `verify-source` — link bot-blocked or undated; confirm manually.
- `superseded` — newer edition found; `Replaced-by` holds the swap.
- `keep` — intentional editorial choice (e.g. a backlink we granted); do NOT flag on quality/age grounds. Only revisit if the URL 404s.
- `needs-fix` — maintenance found a problem (wrong figure, doesn't belong, etc.); the Note states the required action. Resolve on the next content pass.

## Regenerating `Referenced-by` (do NOT hand-edit it)

`Referenced-by` is auto-generated — run it before each refresh so it never drifts:

```
python3 scripts/sync_stat_bank_refs.py            # --check (dry run; exits 1 on drift)
python3 scripts/sync_stat_bank_refs.py --write     # apply
```

It reads each entry's `Source:` URL, normalises it (drops scheme/www/query/fragment/trailing-slash), greps `content/**/*.md`, and rewrites the list (with `(×N)` when a file cites a source more than once). **Semantic:** `Referenced-by` = "pages that cite this *source URL*," not "pages making this exact claim." When several entries share one URL (e.g. the three Gartner `b2b-buying-journey` stats), they all get the same file list — which is what you want: if that page changes, every entry and every listed file needs review.
